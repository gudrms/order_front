# 관리자 계정 관리 시나리오 기획안

## 1. 배경

관리자 웹은 내부 운영 도구다. 현재 매장 규모와 운영 방식에서는 점주가 직접 회원가입하고 이메일 인증을 완료한 뒤 초대코드로 매장 권한을 받는 셀프서비스 모델이 과하다. Supabase 기본 SMTP 발송 한도와 도달 불안정도 있어 실제 운영에서 가입 완료가 막힐 수 있다.

따라서 관리자 계정은 마스터 관리자(`ADMIN`)가 직접 만들고, 역할과 매장을 함께 지정한다. 점주는 발급받은 이메일과 초기 비밀번호로 로그인만 한다.

## 2. 목표

- 외부 자가 회원가입을 제거한다.
- 무인증 `POST /auth/register` 보안 구멍을 제거한다.
- 이메일 인증, `/auth/callback`, `/setup`, 초대코드 입력 흐름을 폐기한다.
- `ADMIN`이 admin UI에서 계정을 생성, 조회, 비밀번호 초기화, 삭제할 수 있게 한다.
- `OWNER` 계정은 지정된 매장만 운영할 수 있게 한다.
- 기존 `ADMIN` 마스터 계정은 이미 존재한다고 보고 별도 시드 작업은 하지 않는다.

## 3. 역할

| 역할 | 설명 | 접근 범위 |
|---|---|---|
| `ADMIN` | 마스터 관리자 | 전체 매장, 브랜드 메뉴, 계정 관리, 운영 관리 |
| `OWNER` | 점주/매장 관리자 | 본인에게 연결된 매장 운영 |
| `USER` | 일반 고객 또는 권한 미부여 계정 | 관리자 웹 접근 불가 |

관리자 웹에서 정상 운영 가능한 역할은 `ADMIN`, `OWNER`만이다.

## 4. 계정 생성 정책

### 입력값

| 필드 | 필수 | 설명 |
|---|---:|---|
| 이메일 | 예 | 로그인 ID. 실제 사용하는 이메일 주소 |
| 초기 비밀번호 | 예 | 마스터가 직접 입력. 8자 이상 |
| 이름 | 선택 | 점주 또는 관리자 이름 |
| 전화번호 | 선택 | 운영 연락용 |
| 역할 | 예 | `ADMIN` 또는 `OWNER` |
| 매장 | `OWNER`일 때 예 | 해당 계정이 관리할 매장 |

### 생성 처리

1. `ADMIN`이 `/accounts` 화면에서 계정 정보를 입력한다.
2. 백엔드는 현재 JWT 사용자가 `ADMIN`인지 확인한다.
3. 백엔드는 Supabase Admin API `createUser`를 호출한다.
4. `email_confirm: true`로 생성해 이메일 인증 메일 발송을 생략한다.
5. 백엔드는 앱 DB `User` 행을 생성한다.
6. 역할이 `OWNER`이면 선택한 `Store.ownerId`에 생성된 사용자 ID를 연결한다.
7. 생성 완료 후 계정 목록을 갱신한다.

실패 시 Supabase Auth 유저만 생성되고 DB 생성이 실패하는 중간 상태를 피해야 한다. DB 처리 실패 시 생성된 Supabase Auth 유저를 삭제한다.

## 5. 화면 시나리오

### A. 로그인

1. 사용자가 `/login`에 접속한다.
2. 화면에는 이메일, 비밀번호, 로그인 버튼만 표시된다.
3. 회원가입 버튼, 비밀번호 확인, 이메일 인증 안내는 표시하지 않는다.
4. 정상 계정으로 로그인하면 대시보드로 이동한다.
5. `ADMIN`/`OWNER` 권한이 없거나 DB 프로필이 없으면 권한 안내 화면으로 이동한다.

기대 결과:

- 셀프 가입 진입점이 없다.
- 잘못된 계정은 로그인 실패 메시지를 본다.
- 권한 없는 계정은 관리자 기능을 사용할 수 없다.

### B. 계정 목록

1. `ADMIN`이 사이드바의 `계정 관리`로 이동한다.
2. 계정 목록에는 `ADMIN`, `OWNER` 계정만 표시된다.
3. 각 행에는 이메일, 이름, 역할, 연결 매장, 생성일이 표시된다.
4. `OWNER`는 이 화면에 접근할 수 없다.

기대 결과:

- `ADMIN`은 전체 관리자 계정을 확인할 수 있다.
- `OWNER`가 `/accounts`에 직접 접근하면 대시보드 또는 권한 차단 화면으로 이동한다.

### C. 점주 계정 생성

1. `ADMIN`이 이메일, 초기 비밀번호, 이름, 전화번호를 입력한다.
2. 역할을 `OWNER`로 선택한다.
3. 관리할 매장을 선택한다.
4. `계정 생성`을 누른다.
5. 생성된 계정이 목록에 표시된다.
6. 선택한 매장의 `ownerId`가 새 계정 ID로 연결된다.

기대 결과:

- 점주는 별도 회원가입 없이 전달받은 이메일/비밀번호로 로그인 가능하다.
- 점주는 본인 매장 주문, 메뉴, 매장 설정만 접근 가능하다.
- 다른 매장 데이터는 조회하거나 수정할 수 없다.

### D. 마스터 관리자 계정 생성

1. `ADMIN`이 역할을 `ADMIN`으로 선택한다.
2. 매장 선택 없이 계정을 생성한다.
3. 생성된 계정은 전체 관리자 권한을 가진다.

기대 결과:

- 새 `ADMIN`은 계정 관리, 브랜드 메뉴, 전체 매장 관리에 접근 가능하다.
- 매장 선택은 필수값이 아니다.

### E. 비밀번호 초기화

1. `ADMIN`이 계정 목록에서 특정 계정의 새 비밀번호를 입력한다.
2. `비밀번호 초기화` 버튼을 누른다.
3. 백엔드는 Supabase Admin API `updateUserById`로 비밀번호를 변경한다.
4. 해당 사용자는 새 비밀번호로 로그인한다.

기대 결과:

- 이메일 발송 없이 즉시 비밀번호가 변경된다.
- 8자 미만 비밀번호는 프론트와 백엔드에서 모두 거부한다.

### F. 계정 삭제

1. `ADMIN`이 계정 목록에서 삭제 버튼을 누른다.
2. 확인 모달에서 삭제를 확정한다.
3. 백엔드는 Supabase Admin API `deleteUser`를 호출한다.
4. 앱 DB `User` 행을 삭제한다.
5. 해당 계정이 `Store.ownerId`에 연결되어 있으면 `ownerId`를 `null`로 해제한다.

기대 결과:

- 삭제된 계정은 더 이상 로그인할 수 없다.
- 연결 매장은 남아 있지만 점주 연결은 해제된다.
- 자기 자신 삭제는 막는다.

## 6. 백엔드 API

Base path: `/api/v1/admin/accounts`

| Method | Path | 권한 | 설명 |
|---|---|---|---|
| `GET` | `/admin/accounts` | `ADMIN` | 관리자 계정 목록 |
| `POST` | `/admin/accounts` | `ADMIN` | 계정 생성 |
| `PATCH` | `/admin/accounts/:userId/password` | `ADMIN` | 비밀번호 초기화 |
| `DELETE` | `/admin/accounts/:userId` | `ADMIN` | 계정 삭제 |

공통 요구사항:

- 모든 엔드포인트는 Supabase JWT 인증이 필요하다.
- 앱 DB의 현재 사용자 역할이 `ADMIN`이어야 한다.
- Supabase Admin API 호출은 백엔드에서만 수행한다.
- `SUPABASE_SERVICE_KEY`는 브라우저에 노출하지 않는다.

## 7. 제거 대상

| 대상 | 처리 |
|---|---|
| 로그인 화면 회원가입 모드 | 제거 |
| `supabase.auth.signUp()` 호출 | admin 앱에서 제거 |
| `/auth/callback` | 제거 |
| `/setup` | 제거 |
| `/pending` 초대코드 입력 | 권한 안내 화면으로 단순화 |
| 무인증 `POST /auth/register` | 제거 |
| 초대코드 기반 OWNER 승격 | 제거 |
| `Store.inviteCode` | Prisma 마이그레이션 시 제거 검토 |
| `refreshInviteCode` API/UI | 제거 |

`Store.inviteCode` 컬럼 제거는 DB 마이그레이션이 필요하므로 계정 관리 전환 1차 작업과 분리한다.

## 8. 예외 및 검증

### 생성 실패

- 이미 존재하는 이메일이면 생성 실패 메시지를 표시한다.
- `OWNER`인데 매장을 선택하지 않으면 생성 버튼 또는 백엔드에서 막는다.
- 매장 ID가 존재하지 않으면 404로 응답한다.
- Supabase 생성 후 DB 저장 실패 시 Supabase 유저를 롤백 삭제한다.

### 권한 실패

- `OWNER`가 `/admin/accounts` API를 호출하면 403.
- DB에 사용자 행이 없는 Supabase 세션은 관리자 권한 없음으로 처리한다.
- `USER` 역할은 관리자 웹 기능 접근 불가.

### 삭제 실패

- 자기 자신 삭제는 400.
- 없는 계정 삭제는 404.
- Supabase 삭제 실패 시 DB 삭제도 진행하지 않는다.

## 9. 자동화 테스트 후보

### Backend unit

- `ADMIN`만 계정 목록을 조회할 수 있다.
- `OWNER` 생성 시 Supabase `createUser`와 DB `User` 생성, `Store.ownerId` 연결이 호출된다.
- `ADMIN` 생성 시 매장 연결 없이 `User.role = ADMIN`으로 저장된다.
- DB 저장 실패 시 Supabase `deleteUser` 롤백이 호출된다.
- 비밀번호 초기화는 `updateUserById`를 호출한다.
- 자기 자신 삭제는 거부한다.

### Admin E2E

- 로그인 페이지에 회원가입 버튼이 없다.
- `ADMIN` 로그인 후 사이드바에 `계정 관리`가 보인다.
- `OWNER` 로그인 후 `계정 관리`가 보이지 않는다.
- 계정 생성 폼에서 `OWNER` 선택 시 매장 선택이 필수다.
- 계정 생성 성공 후 목록에 새 계정이 보인다.
- 비밀번호 초기화 성공 메시지가 표시된다.
- 계정 삭제 후 목록에서 사라진다.

## 10. 구현 순서

1. 로그인 화면에서 회원가입 모드 제거.
2. 백엔드 `POST /auth/register` 제거.
3. `AuthService`에서 초대코드 기반 register 로직 제거.
4. `/auth/callback`, `/setup` 제거.
5. `/pending`은 권한 안내 화면으로 단순화.
6. 백엔드 `AdminAccountsModule` 추가.
7. admin UI `계정 관리` 화면 추가.
8. 사이드바에 `ADMIN` 전용 메뉴 추가.
9. 테스트 시나리오 문서와 E2E 기대값 갱신.
10. 이후 DB 마이그레이션으로 `Store.inviteCode` 제거 여부 결정.
