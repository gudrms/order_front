# Toss POS 플러그인 테스트 시나리오

## 목적

배달 주문, 결제, 메뉴 동기화, 취소 이벤트가 Toss POS와 백엔드 사이에서 끊기지 않고 동작하는지 검증한다.

## 사전 조건

- Toss POS 개발자 환경에서 플러그인이 설치되어 있다.
- `PLUGIN_API_URL`이 POS 기기에서 접근 가능한 HTTPS 주소다.
- `PLUGIN_POS_API_KEY`와 백엔드 `POS_INTEGRATION_API_KEY`가 일치한다.
- `PLUGIN_SUPABASE_URL`, `PLUGIN_SUPABASE_ANON_KEY`, `PLUGIN_STORE_ID`가 설정되어 있다.
- Supabase Realtime에서 주문 테이블 변경 이벤트가 발행된다.
- 테스트 매장 메뉴가 Toss POS와 백엔드에 매핑되어 있다.

## 시나리오 A. 결제 완료 주문 POS 등록

1. 배달 고객 앱에서 주문을 생성한다.
2. Toss 테스트 결제를 완료한다.
3. 주문이 `PAID` 상태가 되는지 확인한다.
4. 플러그인이 Realtime 또는 polling으로 주문을 감지한다.
5. 플러그인이 Toss POS 주문과 결제를 생성한다.
6. 백엔드에 POS 등록 성공 PATCH를 보낸다.
7. POS 화면과 DB 상태를 확인한다.

기대 결과:

- POS 화면에 주문 번호, 메뉴, 금액이 표시된다.
- 결제 완료 상태로 표시된다.
- DB 주문 상태가 `CONFIRMED` 또는 운영 정책에 맞는 POS 등록 완료 상태가 된다.
- `tossOrderId`가 저장된다.
- 같은 주문이 중복 등록되지 않는다.

## 시나리오 B. 배달 앱 또는 관리자 취소의 POS 반영

1. POS 등록이 완료된 주문을 준비한다.
2. 관리자 화면 또는 고객 앱에서 주문을 취소한다.
3. 플러그인이 주문 취소 이벤트를 감지한다.
4. Toss POS 주문 취소 API를 호출한다.
5. POS 화면에서 취소 상태를 확인한다.

기대 결과:

- DB의 `CANCELLED` 상태가 POS에 반영된다.
- 이미 취소된 POS 주문에 대한 재시도는 치명 오류로 이어지지 않는다.
- `tossOrderId`가 없는 주문 취소는 POS 취소 요청을 보내지 않는다.

## 시나리오 C. POS 결제 취소의 백엔드 반영

1. POS 등록이 완료된 주문을 준비한다.
2. POS 화면에서 결제를 취소한다.
3. 플러그인의 결제 취소 이벤트 콜백을 확인한다.
4. 백엔드에서 `tossOrderId`로 주문을 찾는다.
5. 백엔드 주문 상태를 `CANCELLED`로 변경한다.

기대 결과:

- POS 결제 취소가 10초 이내 백엔드 주문 상태에 반영된다.
- 주문을 찾지 못한 경우 오류 로그가 남고 무한 재시도하지 않는다.
- 관리자 화면에서 취소 상태를 확인할 수 있다.

## 시나리오 D. 결제 미완료 주문 타임아웃

1. 고객 앱에서 주문 생성까지만 진행하고 결제는 완료하지 않는다.
2. 결제 타임아웃 작업을 기다리거나 수동 실행한다.
3. 주문 상태와 POS 화면을 확인한다.

기대 결과:

- 결제 미완료 주문은 `CANCELLED`가 된다.
- POS에는 주문이 등록되지 않는다.
- 플러그인은 `tossOrderId`가 없는 취소 이벤트를 무시한다.

## 시나리오 E. 메뉴 동기화

1. Toss POS에서 메뉴를 추가한다.
2. 플러그인 메뉴 동기화가 실행되는지 확인한다.
3. 백엔드 메뉴와 옵션을 확인한다.
4. Toss POS에서 메뉴 가격, 품절 상태, 옵션을 수정한다.
5. Toss POS에서 메뉴를 삭제한다.

기대 결과:

- 새 메뉴, 옵션 그룹, 옵션이 백엔드에 생성된다.
- 가격과 품절 상태 변경이 반영된다.
- 삭제된 메뉴는 물리 삭제 대신 비활성 처리된다.
- 빠른 연속 수정은 debounce되어 과도한 API 호출이 발생하지 않는다.

자동화 후보:

- `apps/toss-pos-plugin/src/__tests__/*.test.ts`
- 실제 POS 기기 연동은 수동 검증 유지

## 시나리오 F. 장애와 재시도

1. 백엔드 API를 일시적으로 중단한다.
2. 결제 완료 주문을 생성한다.
3. 플러그인 재시도 로그를 확인한다.
4. 백엔드 API를 복구한다.
5. 다음 polling 또는 Realtime 처리 결과를 확인한다.

기대 결과:

- 플러그인이 재시도 후 실패 상태를 기록한다.
- POS에 등록만 되고 백엔드 반영에 실패한 주문은 정리 또는 재처리 정책을 따른다.
- 복구 후 같은 주문이 무한 중복 생성되지 않는다.
- 운영자가 확인할 수 있는 로그가 남는다.
