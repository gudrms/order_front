# 백엔드 API 테스트 시나리오

## 목적

프론트엔드, 관리자, POS 플러그인이 의존하는 API 계약과 상태 전이를 검증한다.

## 사전 조건

- 백엔드 API가 실행 중이다.
- Supabase PostgreSQL과 Auth 설정이 준비되어 있다.
- Prisma Client가 생성되어 있다.
- 테스트 매장, 사용자, 메뉴, 쿠폰 데이터가 준비되어 있다.
- 인증이 필요한 API는 Supabase JWT를 사용한다.

## 시나리오 A. 헬스 체크와 공통 응답

1. `GET /health` 또는 배포 헬스 체크 엔드포인트를 호출한다.
2. 공개 API 하나를 호출한다.
3. 존재하지 않는 리소스를 호출한다.

기대 결과:

- 헬스 체크는 200을 반환한다.
- 성공 응답은 `{ statusCode, data }` 형태를 유지한다.
- 오류 응답은 `{ statusCode, timestamp, path, message }` 형태를 유지한다.

자동화 후보:

- 백엔드 API smoke test

## 시나리오 B. 인증과 권한

1. 인증 없이 보호 API를 호출한다.
2. 만료되었거나 잘못된 JWT로 호출한다.
3. 정상 고객 JWT로 고객 API를 호출한다.
4. 고객 JWT로 관리자 API를 호출한다.
5. 관리자 JWT로 본인 매장 API를 호출한다.

기대 결과:

- 인증 없는 요청은 401이다.
- 잘못된 JWT는 401이다.
- 권한이 없는 매장 접근은 403이다.
- 본인 매장 관리자 요청은 성공한다.

## 시나리오 C. 관리자 계정 관리와 매장 생성 API

1. 기존 `ADMIN` 권한 계정으로 인증한다.
2. `POST /api/v1/stores`를 호출해 새 매장을 생성한다.
3. 같은 `storeType + branchId`로 중복 생성을 시도한다.
4. `POST /api/v1/admin/accounts`로 `OWNER` 계정을 생성하고 매장을 연결한다.
5. `GET /api/v1/admin/accounts`로 생성된 계정과 연결 매장을 확인한다.
6. `PATCH /api/v1/admin/accounts/{userId}/password`로 비밀번호를 초기화한다.
7. `DELETE /api/v1/admin/accounts/{userId}`로 계정을 삭제한다.
8. `GET /api/v1/stores/me`로 삭제 또는 연결 해제 후 매장 접근 권한을 확인한다.

기대 결과:

- `ADMIN` 권한이 없는 계정은 새 매장을 생성할 수 없다.
- 중복 `storeType + branchId` 생성은 실패한다.
- `ADMIN` 권한이 없는 계정은 `/admin/accounts` API를 호출할 수 없다.
- 생성된 `OWNER`는 해당 매장의 `ownerId`로 연결된다.
- 비밀번호 초기화는 Supabase Admin API를 통해 이메일 발송 없이 처리된다.
- 계정 삭제 시 매장의 `ownerId`는 해제된다.

## 시나리오 D. 테이블 주문 API

1. 매장 카테고리와 메뉴를 조회한다.
2. 첫 주문 API로 테이블 세션과 주문을 생성한다.
3. 같은 테이블에 추가 주문을 생성한다.
4. 현재 세션 API로 주문 목록과 합계를 조회한다.
5. 세션 완료 API를 호출한다.

기대 결과:

- 활성 메뉴와 옵션만 주문 가능하다.
- 첫 주문은 새 세션을 생성한다.
- 추가 주문은 기존 활성 세션에 연결된다.
- 세션 완료 후 현재 세션 조회 결과가 바뀐다.
- 금액 합계가 주문 항목과 옵션 금액을 정확히 반영한다.

연결 문서:

- `docs/test-scenarios/table-order.md`

## 시나리오 E. 배달 주문과 결제

Toss POS가 없는 1차 테스트에서도 Toss Payments 결제 모듈을 통한 배달 주문 생성, 결제 승인, 관리자 주문 처리는 검증한다. 제외 대상은 POS 플러그인 주문 등록과 POS 메뉴 동기화다.

1. 고객 JWT로 주문 생성 API를 호출한다.
2. Toss 결제 승인 API를 호출한다.
3. 결제 실패 API를 호출한다.
4. 고객 취소 API를 호출한다.

기대 결과:

- 주문 생성 시 `PENDING_PAYMENT` 상태가 된다.
- 결제 승인 성공 시 결제 상태가 `PAID`가 된다.
- 결제 금액 불일치는 400으로 막힌다.
- 결제 실패와 고객 취소는 주문 상태와 결제 상태를 일관되게 변경한다.

연결 문서:

- `docs/test-scenarios/delivery-customer.md`

## 시나리오 F. 관리자 주문 처리

1. 관리자 JWT로 매장 주문 목록을 조회한다.
2. 주문 상태를 `CONFIRMED`, `COOKING`, `READY`, `COMPLETED` 순서로 변경한다.
3. 배달 상태를 `ASSIGNED`, `DELIVERING`, `DELIVERED` 순서로 변경한다.
4. 허용되지 않는 역방향 상태 변경을 시도한다.

기대 결과:

- 정상 상태 전이는 성공한다.
- 허용되지 않는 상태 전이는 400 또는 정책에 맞는 오류로 막힌다.
- 배달 완료와 주문 완료 상태가 서로 모순되지 않는다.

연결 문서:

- `docs/test-scenarios/admin.md`

## 시나리오 G. Toss 환불

Toss 테스트 결제 또는 결제 모킹이 준비된 뒤 수행한다.

1. 관리자 JWT로 Toss 전액 환불 API를 호출한다.
2. 부분 환불 API를 호출한다.
3. 남은 금액보다 큰 환불을 시도한다.
4. 이미 전액 환불된 주문을 다시 환불한다.

기대 결과:

- 전액 환불 후 결제 상태는 `REFUNDED`다.
- 부분 환불 후 결제 상태와 잔여 금액이 정확하다.
- 초과 환불과 중복 환불은 실패한다.
- Toss API 오류는 내부 상태를 불완전하게 변경하지 않는다.

## 시나리오 H. POS 연동 API

Toss POS 플러그인 검증 단계에서 수행한다. Toss 없이 1차 수동 테스트를 진행할 때는 건너뛴다.

1. `x-pos-api-key` 없이 POS API를 호출한다.
2. 정상 `x-pos-api-key`로 대기 주문 목록을 조회한다.
3. POS 등록 성공 상태를 PATCH한다.
4. 같은 `Idempotency-Key`로 같은 요청을 재시도한다.
5. POS 등록 실패 API를 호출한다.

기대 결과:

- API 키가 없거나 틀리면 401 또는 403이다.
- 대기 주문 목록에는 POS로 넘겨야 하는 주문만 포함된다.
- 등록 성공 후 주문 상태와 `tossOrderId`가 반영된다.
- 멱등 요청은 중복 상태 변경을 만들지 않는다.
- 등록 실패 횟수와 실패 메시지가 기록된다.

연결 문서:

- `docs/test-scenarios/toss-pos-plugin.md`

## 시나리오 I. 메뉴와 쿠폰

1. 관리자 JWT로 카테고리, 메뉴, 옵션 그룹, 옵션을 생성한다.
2. 메뉴 가격과 품절 상태를 수정한다.
3. 쿠폰을 생성하고 고객에게 발급한다.
4. 고객 JWT로 쿠폰을 등록하고 주문에 적용한다.

기대 결과:

- 메뉴 구조가 주문 앱 조회 결과에 반영된다.
- 필수 옵션 정책이 주문 생성 API에서 검증된다.
- 쿠폰 최소 주문 금액, 만료일, 중복 사용 정책이 지켜진다.
