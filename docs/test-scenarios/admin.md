# 관리자 웹 테스트 시나리오

## 목적

매장 관리자가 최초 로그인부터 매장 생성, 메뉴 등록, 주문 운영까지 안정적으로 처리할 수 있는지 검증한다. Toss POS 연동이 없는 초기 운영은 `ADMIN_DIRECT` 메뉴 관리 모드를 기준으로 검증한다. 배달 주문 결제는 Toss Payments 결제 모듈로 진행하고, POS 연동 없이 관리자 페이지에서 접수/배달 상태를 처리한다.

## 사전 조건

- 백엔드 API가 실행 중이다.
- 관리자 웹이 실행 중이다.
- 관리자 계정이 존재한다.
- 새 매장 생성 검증 시 계정은 `ADMIN` 권한을 가진다.
- 기존 매장 운영 검증 시 계정은 `ADMIN` 또는 매장 `OWNER` 권한을 가진다.
- Toss POS 없이 테스트할 경우 매장 메뉴 관리 모드는 `ADMIN_DIRECT`를 사용한다.
- Toss POS가 없어도 Toss Payments 결제 모듈을 통한 배달 주문은 테스트한다.
- 주문 운영 검증 전에는 테스트 매장에 메뉴, 카테고리, 주문 데이터가 존재한다.
- Toss 결제 환불 검증은 Toss 테스트 결제 또는 환불 가능한 결제가 준비된 경우에만 수행한다.

## 시나리오 A. 관리자 로그인과 권한

1. 관리자 로그인 페이지에 접속한다.
2. 정상 관리자 계정으로 로그인한다.
3. 프로필이 없는 신규 계정이면 `/setup` 화면으로 이동한다.
4. 이름과 휴대폰 번호를 입력한다.
5. 초대 코드가 있으면 입력하고, 없으면 비워 둔다.
6. 대시보드 또는 주문 화면에 진입한다.
7. 권한이 없는 계정으로 같은 경로에 접근한다.

기대 결과:

- 정상 계정은 매장 선택 또는 기본 매장 화면으로 진입한다.
- 신규 계정은 프로필 설정 완료 후 관리자 화면으로 이동한다.
- 초대 코드를 입력한 계정은 해당 매장 관리자 권한을 얻는다.
- 권한이 없는 계정은 접근 차단 또는 권한 안내 화면을 본다.
- 로그인 세션 새로고침 후에도 권한 상태가 유지된다.

자동화 후보:

- `e2e/admin/auth.spec.ts`

## 시나리오 B. Toss POS 없이 새 매장 시작하기

1. `ADMIN` 권한 계정으로 로그인한다.
2. `/store` 화면에 진입한다.
3. 연결된 매장이 없다면 `새 매장 만들기`를 누른다.
4. 매장 타입, 지점 ID, 매장명, 지점명, 전화번호, 주소를 입력한다.
5. 매장을 생성한다.
6. 생성된 매장을 선택한다.
7. 메뉴 관리 방식을 `ADMIN_DIRECT`로 변경한다.
8. 배달을 받을 매장이라면 배달 주문 접수를 `ON`으로 켠다.
9. 최소 주문 금액, 배달비, 무료 배달 기준, 예상 배달 시간을 입력한다.
10. 저장한다.
11. 초대 코드가 필요한 경우 초대 코드를 재발급한다.

기대 결과:

- 새 매장이 생성되고 매장 목록에 표시된다.
- 생성 직후 선택된 매장의 기본 정보가 `/store` 화면에 표시된다.
- `ADMIN_DIRECT` 설정이 저장된다.
- 배달 설정값이 새로고침 후에도 유지된다.
- 발급된 초대 코드로 다른 관리자 계정이 해당 매장에 합류할 수 있다.
- Toss POS 연동이 없어도 메뉴를 직접 등록할 수 있는 상태가 된다.

수동 체크 포인트:

- `storeType`과 `branchId`는 고객 주문 URL에 쓰이므로 영문 소문자/숫자/하이픈 기준으로 입력한다.
- 같은 `storeType + branchId` 조합은 중복 생성되지 않아야 한다.
- Toss POS 없이 시작하는 매장은 `TOSS_POS`가 아니라 `ADMIN_DIRECT`여야 한다.

## 시나리오 C. 초기 메뉴 등록

1. `/menu` 화면에 진입한다.
2. 매장이 `ADMIN_DIRECT` 모드인지 확인한다.
3. 첫 카테고리를 생성한다.
4. 첫 메뉴를 생성한다.
5. 메뉴 가격, 설명, 이미지 URL을 입력한다.
6. 필수 옵션 그룹을 생성한다.
7. 옵션 항목을 추가한다.
8. 고객 앱 또는 테이블 주문 앱에서 메뉴가 노출되는지 확인한다.
9. 메뉴를 품절 처리한 뒤 주문 앱에서 주문이 막히는지 확인한다.

기대 결과:

- 카테고리와 메뉴가 관리자 화면에 표시된다.
- 필수 옵션 정책이 저장된다.
- 고객 주문 화면에 활성 메뉴만 노출된다.
- 품절 메뉴는 보이더라도 주문할 수 없거나 품절로 표시된다.
- 메뉴 가격 변경 후 새 주문에는 변경된 가격이 적용된다.

자동화 후보:

- `e2e/admin/menu.spec.ts`

## 시나리오 D. 주문 상태 변경

1. `/orders` 화면에 진입한다.
2. `PAID` 또는 `PENDING` 상태 주문을 확인한다.
3. 주문 상태 액션을 눌러 `CONFIRMED`로 변경한다.
4. 같은 주문을 `COOKING`, `READY`, `COMPLETED` 순서로 진행한다.
5. 화면을 새로고침한다.

기대 결과:

- 각 상태 변경 요청이 성공한다.
- 주문 행과 상세 패널의 상태가 동일하게 갱신된다.
- 새로고침 후에도 마지막 상태가 유지된다.
- 이미 완료된 주문에는 부적절한 다음 액션이 노출되지 않는다.

자동화 후보:

- `e2e/admin/orders.spec.ts`

## 시나리오 E. 배달 상태 변경

1. 배달 주문을 준비한다.
2. 관리자 주문 화면에서 배달 상태 액션을 누른다.
3. `PENDING`에서 `ASSIGNED`로 변경한다.
4. 다음 액션으로 `DELIVERING`, `DELIVERED`까지 진행한다.

기대 결과:

- 배달 상태가 단계별로 변경된다.
- `DELIVERED` 처리 시 주문 상태가 완료 상태와 충돌하지 않는다.
- 고객 주문 내역 또는 배달 고객 앱에서 상태가 반영된다.

자동화 후보:

- `e2e/admin/orders.spec.ts`
- `docs/test-scenarios/full-flow.md`의 배달 완료 흐름

## 시나리오 F. Toss 결제 환불

토스가 없는 초기 테스트에서는 이 시나리오를 건너뛴다. Toss 결제 연동 또는 테스트 결제 데이터가 준비된 뒤 수행한다.

1. Toss 결제가 완료된 주문을 준비한다.
2. 전액 환불을 선택하고 환불 사유를 입력한다.
3. 결제 상태와 주문 상태를 확인한다.
4. 부분 환불 가능한 주문에서 환불 금액과 사유를 입력한다.
5. 이미 환불된 금액보다 큰 금액으로 부분 환불을 시도한다.

기대 결과:

- 전액 환불 성공 시 결제 상태는 `REFUNDED`, 주문 상태는 `CANCELLED`가 된다.
- 부분 환불 성공 시 남은 결제 금액과 환불 누적 금액이 정확하다.
- 잘못된 환불 금액은 서버 검증 오류로 막힌다.
- Toss API 실패 시 운영자가 이해할 수 있는 오류 메시지가 표시된다.

자동화 후보:

- `e2e/admin/orders.spec.ts`
- 백엔드 결제 서비스 단위 테스트

## 시나리오 G. 메뉴 관리 모드

### ADMIN_DIRECT

1. 매장 메뉴 관리 모드가 `ADMIN_DIRECT`인지 확인한다.
2. 새 카테고리를 생성한다.
3. 새 메뉴를 생성한다.
4. 메뉴 가격, 노출 상태, 품절 상태, 옵션 그룹을 수정한다.
5. 고객 앱 또는 테이블 주문 앱에서 메뉴 노출을 확인한다.

기대 결과:

- 생성한 카테고리와 메뉴가 즉시 목록에 반영된다.
- 가격과 품절 상태가 주문 화면에 반영된다.
- 필수 옵션이 빠진 주문은 생성되지 않는다.

### TOSS_POS

1. 매장 메뉴 관리 모드가 `TOSS_POS`인지 확인한다.
2. Toss 메뉴 동기화 버튼을 누른다.
3. 동기화 로그와 요약 카운트를 확인한다.
4. 동기화된 메뉴가 주문 앱에 노출되는지 확인한다.

기대 결과:

- 동기화 성공 메시지와 수신/생성/갱신 수량이 표시된다.
- 중복 동기화해도 메뉴가 중복 생성되지 않는다.
- Toss POS에서 품절 처리한 메뉴가 고객 주문 화면에도 반영된다.

자동화 후보:

- `e2e/admin/menu.spec.ts`

## 시나리오 H. 매장 운영 설정

1. `/store` 화면에 진입한다.
2. 배달 가능 여부, 최소 주문 금액, 배달비, 영업 상태를 변경한다.
3. 저장 후 새로고침한다.
4. 고객 주문 앱에서 변경된 설정이 반영되는지 확인한다.

기대 결과:

- 저장 성공 메시지가 표시된다.
- 새로고침 후 변경값이 유지된다.
- 영업 중지 또는 배달 중지 상태에서는 고객 주문 진입이 제한된다.

자동화 후보:

- `e2e/admin/store.spec.ts`
