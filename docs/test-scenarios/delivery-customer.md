# 배달 고객 앱 테스트 시나리오

## 목적

고객이 배달 주문을 생성하고 Toss 결제를 완료하며, 주문 내역과 상태 변경을 확인할 수 있는지 검증한다.

## 사전 조건

- 배달 고객 앱이 실행 중이다.
- 백엔드 API가 실행 중이다.
- 테스트 매장이 활성화되어 있고 배달 가능 상태다.
- 테스트 메뉴가 존재한다.
- Toss 테스트 결제 키 또는 결제 모킹 환경이 준비되어 있다.
- 로그인 검증 시 Supabase Auth 테스트 계정이 존재한다.

## Toss POS 없는 1차 테스트 범위

Toss POS 플러그인이 없어도 배달 고객 앱은 Toss Payments 결제 모듈로 결제 완료까지 검증한다. 여기서 제외하는 것은 POS 플러그인 주문 등록과 POS 메뉴 동기화다.

1. 메인 화면 진입
2. 로그인 화면 진입
3. 메뉴 목록 표시
4. 메뉴 상세와 옵션 선택
5. 장바구니 담기
6. 배달 주소와 연락처 입력
7. 체크아웃 화면에서 최종 결제 금액 확인
8. Toss Payments 결제 위젯으로 테스트 결제 완료
9. 주문 완료 화면과 주문 번호 확인
10. 관리자 주문 화면에서 배달 주문 확인

기대 결과:

- 메뉴와 장바구니 금액이 정상 계산된다.
- 배달 설정이 꺼진 매장은 주문 진행이 제한된다.
- 최소 주문 금액 미만이면 주문 진행이 제한된다.
- Toss Payments 결제 성공 후 주문 상태와 결제 상태가 관리자 화면에 반영된다.
- Toss POS 플러그인이 없어도 관리자 페이지에서 배달 접수/배정/완료 처리가 가능하다.

## 시나리오 A. 기본 페이지와 비로그인 접근

1. `/`에 접속한다.
2. 하단 내비게이션과 주요 콘텐츠가 표시되는지 확인한다.
3. `/orders`에 비로그인 상태로 접속한다.
4. 로그인 안내 버튼을 누른다.

기대 결과:

- 메인 화면이 정상 렌더링된다.
- 비로그인 주문 내역 접근 시 로그인 필요 안내가 표시된다.
- 로그인 버튼 클릭 시 `/login`으로 이동한다.

자동화 후보:

- `e2e/delivery-customer/pages.spec.ts`

## 시나리오 B. 로그인

1. `/login`에 접속한다.
2. 카카오 로그인 버튼이 표시되는지 확인한다.
3. Apple 로그인 버튼이 표시되는지 확인한다.
4. 로그인 없이 둘러보기 링크를 누른다.
5. 실제 로그인 계정으로 로그인한다.

기대 결과:

- 소셜 로그인 버튼과 비로그인 둘러보기가 표시된다.
- 로그인 성공 후 사용자 정보가 동기화된다.
- 로그인 후 주문 내역 화면 접근이 가능하다.

자동화 후보:

- 기본 UI는 Playwright로 자동화
- 실제 OAuth는 스테이징 수동 검증 또는 인증 모킹으로 분리

## 시나리오 C. 메뉴 선택과 장바구니

1. `/menu`에 접속한다.
2. 매장 메뉴와 카테고리를 확인한다.
3. 메뉴 상세를 열고 옵션을 선택한다.
4. 장바구니에 담는다.
5. 수량을 변경한다.

기대 결과:

- 메뉴 목록이 정상 표시된다.
- 필수 옵션 누락 시 장바구니 담기가 막힌다.
- 장바구니 합계가 메뉴 가격, 옵션 가격, 수량을 반영한다.
- 품절 메뉴는 주문할 수 없다.

## 시나리오 D. 배달 주문 생성

1. 장바구니에 메뉴를 담는다.
2. 배달 주소, 상세 주소, 연락처, 요청사항을 입력한다.
3. 쿠폰이 있다면 적용한다.
4. 주문 생성 버튼을 누른다.

기대 결과:

- 서버에 주문 생성 요청이 전송된다.
- 주문 상태는 `PENDING_PAYMENT`로 시작한다.
- 결제에 사용할 `payment.orderId`와 금액이 반환된다.
- 쿠폰 적용 금액과 최종 결제 금액이 일치한다.

## 시나리오 E. Toss 결제 성공

Toss 테스트 결제 키 또는 결제 모킹이 준비된 뒤 수행한다.

1. 주문 생성 후 Toss 결제창을 연다.
2. 테스트 결제를 완료한다.
3. 결제 성공 콜백으로 돌아온다.
4. 주문 완료 화면과 주문 번호를 확인한다.
5. 주문 내역에서 해당 주문을 확인한다.

기대 결과:

- 결제 승인 API가 성공한다.
- 결제 상태는 `PAID`가 된다.
- 주문 상태는 관리자 또는 POS 연동 흐름으로 이어질 수 있는 상태가 된다.
- 관리자 주문 화면과 POS 플러그인에서 같은 주문을 볼 수 있다.

## 시나리오 F. Toss 결제 실패와 취소

Toss 테스트 결제 키 또는 결제 모킹이 준비된 뒤 수행한다.

1. 결제창에서 사용자가 결제를 취소한다.
2. 결제 실패 콜백으로 돌아온다.
3. 주문 상태를 확인한다.
4. 같은 장바구니로 다시 결제를 시도한다.

기대 결과:

- 실패 사유가 사용자에게 표시된다.
- 서버의 결제 실패 처리 API가 호출된다.
- 실패한 주문은 운영 정책에 따라 `CANCELLED` 또는 재시도 가능한 상태가 된다.
- 재결제 시 금액과 주문 식별자가 충돌하지 않는다.

## 시나리오 G. 주문 취소

1. 결제 완료 주문을 생성한다.
2. 고객 주문 내역에서 취소 가능 상태인지 확인한다.
3. 취소 사유를 입력하고 취소한다.
4. 관리자 화면과 결제 상태를 확인한다.

기대 결과:

- 취소 가능 상태에서만 취소 버튼이 노출된다.
- 취소 성공 시 주문 상태가 `CANCELLED`가 된다.
- 결제 취소 또는 환불 상태가 일관되게 반영된다.
- 이미 조리 중이거나 배달 중인 주문은 고객 취소가 제한된다.
