# 테이블 주문 테스트 시나리오

## 목적

QR로 진입한 매장 내 고객이 테이블 번호를 기준으로 메뉴를 보고 주문을 생성하며, 세션과 추가 주문이 정확히 연결되는지 검증한다.

## 사전 조건

- 테이블 주문 앱이 실행 중이다.
- 백엔드 API가 실행 중이다.
- 테스트 매장 식별자 예: `tacomolly/gimpo`
- 테스트 테이블 번호가 존재한다.
- 테스트 매장에 활성 카테고리와 메뉴가 존재한다.

## 시나리오 A. QR 진입 검증

1. `/tacomolly/gimpo/table/5`에 접속한다.
2. 매장 정보가 정상 조회되는지 확인한다.
3. 메뉴 화면으로 이동하는지 확인한다.
4. `/tacomolly/gimpo/table/abc`처럼 잘못된 테이블 번호로 접속한다.
5. `/tacomolly/gimpo/table/0`처럼 허용되지 않는 번호로 접속한다.

기대 결과:

- 유효한 테이블 번호는 메뉴 화면으로 이동하고 `table=5`가 유지된다.
- 잘못된 테이블 번호는 QR 재스캔 안내 또는 오류 화면을 보여준다.
- 매장 식별자가 잘못되면 주문 가능한 화면으로 진입하지 않는다.

자동화 후보:

- `e2e/table-order/orders.spec.ts`

## 시나리오 B. 메뉴 탐색과 장바구니

1. 메뉴 화면에 진입한다.
2. 카테고리 탭을 변경한다.
3. 메뉴 상세를 연다.
4. 필수 옵션과 선택 옵션을 고른다.
5. 수량을 변경하고 장바구니에 담는다.
6. 장바구니에서 수량을 수정하거나 항목을 삭제한다.

기대 결과:

- 카테고리별 메뉴가 올바르게 필터링된다.
- 품절 메뉴는 주문할 수 없다.
- 필수 옵션을 선택하지 않으면 장바구니 담기가 막힌다.
- 장바구니 합계가 메뉴 가격, 옵션 가격, 수량을 반영한다.

자동화 후보:

- 테이블 주문 Playwright 상세 플로우 추가

## 시나리오 C. 첫 주문 생성

1. 테이블 번호가 있는 메뉴 화면에서 장바구니에 메뉴를 담는다.
2. 주문 확인 화면으로 이동한다.
3. 요청사항을 입력한다.
4. 주문하기를 누른다.
5. 주문 완료 화면과 주문 번호를 확인한다.

기대 결과:

- 서버에 첫 주문 API가 호출된다.
- 새 테이블 세션이 생성된다.
- 주문 상태는 매장 운영 정책에 맞게 `PENDING`, `PAID`, `CONFIRMED` 중 하나로 시작한다.
- 관리자 주문 화면에 같은 주문이 표시된다.

백엔드 확인 포인트:

- `session.id`
- `order.id`
- `tableNumber`
- `items`
- `totalAmount`

## 시나리오 D. 추가 주문

1. 이미 활성 세션이 있는 테이블로 다시 접속한다.
2. 메뉴를 추가로 장바구니에 담는다.
3. 추가 주문을 생성한다.
4. 현재 세션의 주문 목록을 확인한다.

기대 결과:

- 기존 활성 세션에 주문이 연결된다.
- 첫 주문과 추가 주문이 같은 세션 아래에 조회된다.
- 세션 합계 금액이 모든 주문 합계를 반영한다.
- 다른 테이블 세션과 섞이지 않는다.

## 시나리오 E. 세션 완료와 재주문

1. 관리자 또는 API로 테이블 세션을 완료 처리한다.
2. 같은 테이블 QR로 다시 접속한다.
3. 새 주문을 생성한다.

기대 결과:

- 완료된 세션은 더 이상 현재 세션으로 조회되지 않는다.
- 새 주문은 새 세션으로 생성된다.
- 과거 주문 내역과 현재 주문이 섞이지 않는다.

## 시나리오 F. 장애 상황

1. 메뉴 조회 API가 실패하도록 만든다.
2. 주문 생성 API가 실패하도록 만든다.
3. 네트워크가 느린 상태에서 주문 버튼을 연속 클릭한다.

기대 결과:

- 메뉴 조회 실패 시 재시도 또는 오류 안내가 표시된다.
- 주문 생성 실패 시 장바구니가 임의로 비워지지 않는다.
- 중복 클릭으로 같은 주문이 중복 생성되지 않는다.
