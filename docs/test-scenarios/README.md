## 공통 작성 기준

- 목적: 어떤 장애를 막기 위한 검증인지 적는다.
- 사전 조건: 계정, 매장, 메뉴, 결제 키, 서버 상태를 적는다.
- 절차: 사용자가 실제로 누르는 순서 또는 API 호출 순서로 적는다.
- 기대 결과: 화면, DB, API 응답, 외부 연동 결과를 함께 적는다.
- 자동화 후보: Playwright 또는 API 테스트로 옮길 수 있는 범위를 표시한다.

## 테스트 단계

### 1차: Toss POS 없이 직접 검증

- 관리자 로그인과 프로필 설정
- 관리자 새 매장 생성
- 매장 메뉴 관리 방식 `ADMIN_DIRECT` 설정
- 카테고리/메뉴/옵션 직접 등록
- 테이블 QR 진입
- 테이블 첫 주문과 추가 주문
- 관리자 주문 접수, 조리, 완료 상태 변경
- 매장 설정 변경 후 주문 앱 반영 확인
- 배달 고객 앱 메뉴/장바구니/주소 입력
- Toss Payments 결제 모듈을 통한 배달 주문 결제
- 관리자 페이지에서 배달 주문 접수, 배달 배정, 배달 중, 배달 완료 상태 변경

### 2차: Toss POS 연동 준비 후 검증

- 관리자 전액/부분 환불
- Toss POS 플러그인 주문 등록
- POS 취소와 백엔드 상태 반영
- 메뉴 Toss POS 동기화
