# 전체 주문 흐름 테스트 시나리오

## 목적

고객 앱, 테이블 주문, 관리자 웹, 백엔드, Toss POS 플러그인을 함께 사용했을 때 실제 매장 운영 흐름이 끝까지 이어지는지 검증한다.

## 사전 조건

- 백엔드 API가 실행 중이다.
- 관리자 웹, 테이블 주문 앱, 배달 고객 앱이 실행 중이다.
- 테스트 매장은 활성 상태다.
- 메뉴와 옵션이 최소 1개 이상 존재한다.
- 관리자 계정과 고객 테스트 계정이 존재한다.
- Toss POS 없는 1차 테스트에서는 `ADMIN_DIRECT` 메뉴 관리 모드를 사용한다.
- 배달 주문 결제는 Toss Payments 결제 모듈로 검증한다.
- POS 연동까지 검증하는 2차 테스트에서는 Toss POS 플러그인이 준비되어 있다.

## 1차 전체 흐름: Toss POS 없이 관리자, 테이블 주문, 배달 주문 검증

1. 관리자가 로그인한다.
2. 신규 계정이면 프로필 설정을 완료한다.
3. `ADMIN` 계정으로 새 매장을 생성한다.
4. 매장 메뉴 관리 방식을 `ADMIN_DIRECT`로 설정한다.
5. 배달 설정, 최소 주문 금액, 배달비를 저장한다.
6. 카테고리와 메뉴, 필수 옵션을 등록한다.
7. 테이블 QR URL로 진입한다.
8. 메뉴를 선택하고 첫 주문을 생성한다.
9. 관리자 주문 화면에서 주문을 확인한다.
10. 관리자가 주문을 접수, 조리, 준비 완료, 완료 상태로 변경한다.
11. 같은 테이블에서 추가 주문을 생성한다.
12. 현재 세션에 첫 주문과 추가 주문이 같이 묶였는지 확인한다.
13. 배달 고객 앱에서 메뉴를 선택하고 주소를 입력한다.
14. Toss Payments 결제 모듈로 결제를 완료한다.
15. 관리자 주문 화면에서 배달 주문을 확인한다.
16. 관리자가 배달 주문을 접수하고 배달 배정, 배달 중, 배달 완료 상태로 변경한다.

기대 결과:

- 매장 생성부터 메뉴 등록까지 Toss 없이 완료된다.
- 테이블 주문 앱에서 직접 등록한 메뉴가 노출된다.
- 필수 옵션이 누락된 주문은 생성되지 않는다.
- 첫 주문과 추가 주문이 같은 테이블 세션에 연결된다.
- 관리자 주문 상태 변경이 새로고침 후에도 유지된다.
- Toss POS 플러그인이 없어도 배달 주문 결제와 관리자 배달 상태 처리는 완료된다.

## 1차 보류 항목

- Toss POS 주문 등록
- Toss POS 메뉴 동기화

## 시나리오 A. 배달 주문 전체 흐름

Toss 테스트 결제 환경이 준비된 뒤 수행한다.

1. 고객이 배달 앱에서 메뉴를 선택한다.
2. 배달 주소와 연락처를 입력한다.
3. 주문을 생성하고 Toss 결제를 완료한다.
4. 백엔드에서 주문과 결제 상태를 확인한다.
5. Toss POS 플러그인이 주문을 감지해 POS에 등록한다.
6. 관리자 주문 화면에서 신규 주문을 확인한다.
7. 관리자가 주문을 접수하고 조리 상태로 변경한다.
8. 관리자가 배달 배정, 배달 중, 배달 완료 상태로 변경한다.
9. 고객 주문 내역에서 상태 변경을 확인한다.

기대 결과:

- 고객 주문 생성부터 결제 완료까지 금액이 변하지 않는다.
- POS에 같은 주문이 1회만 등록된다.
- 관리자 화면에는 POS 등록 여부와 주문 상태가 일관되게 표시된다.
- 고객 주문 내역은 관리자 상태 변경을 반영한다.
- 최종 상태는 주문 완료와 배달 완료가 모순되지 않는다.

검증 포인트:

- `Order.status`
- `Payment.status`
- `Delivery.status`
- `Order.tossOrderId`
- 관리자 주문 행
- 고객 주문 내역
- POS 주문 화면

## 시나리오 B. 테이블 주문 전체 흐름

1. 고객이 QR로 테이블 주문 앱에 진입한다.
2. 메뉴와 옵션을 선택한다.
3. 첫 주문을 생성한다.
4. 관리자가 주문을 확인하고 조리 상태를 변경한다.
5. 같은 테이블에서 추가 주문을 생성한다.
6. 현재 세션에서 첫 주문과 추가 주문을 확인한다.
7. 관리자가 세션을 완료 처리한다.
8. 같은 테이블 QR로 새 주문을 시작한다.

기대 결과:

- QR의 테이블 번호가 주문과 세션에 정확히 저장된다.
- 추가 주문은 기존 활성 세션에 연결된다.
- 관리자 화면에서 같은 테이블 주문들을 구분할 수 있다.
- 세션 완료 후 새 주문은 새 세션에 연결된다.

검증 포인트:

- `TableSession.status`
- `Order.tableNumber`
- `Order.items`
- 세션 총액
- 관리자 주문 화면

## 시나리오 C. 배달 주문 취소와 환불

1. 고객이 배달 주문을 결제 완료한다.
2. POS 등록과 관리자 표시를 확인한다.
3. 관리자가 전액 환불을 실행한다.
4. 고객 주문 내역을 확인한다.
5. POS 주문 상태를 확인한다.

기대 결과:

- Toss 환불이 성공한다.
- 백엔드 결제 상태가 `REFUNDED`가 된다.
- 주문 상태가 `CANCELLED`가 된다.
- POS 주문도 취소 상태로 반영된다.
- 고객에게 취소 또는 환불 완료 상태가 표시된다.

## 시나리오 D. 결제 실패 복구

1. 고객이 배달 주문을 생성한다.
2. Toss 결제창에서 결제를 취소한다.
3. 고객 앱에서 실패 안내를 확인한다.
4. 같은 메뉴로 다시 주문을 생성하고 결제를 완료한다.
5. 관리자와 POS에 성공 주문만 표시되는지 확인한다.

기대 결과:

- 실패한 주문은 POS에 등록되지 않는다.
- 실패 주문과 성공 주문의 결제 식별자가 충돌하지 않는다.
- 성공 주문만 관리자 운영 흐름으로 들어온다.

## 시나리오 E. 메뉴 변경 후 주문 반영

1. 관리자가 직접 메뉴를 생성하거나 Toss POS 메뉴를 동기화한다.
2. 고객 앱과 테이블 주문 앱에서 새 메뉴가 표시되는지 확인한다.
3. 메뉴 가격을 변경한다.
4. 품절 상태를 켠다.
5. 고객 앱과 테이블 주문 앱에서 변경값을 확인한다.

기대 결과:

- 새 메뉴가 두 주문 채널에 표시된다.
- 가격 변경 후 새 주문 금액이 변경된 가격을 사용한다.
- 품절 메뉴는 주문할 수 없다.
- 기존 주문 금액은 과거 가격 기준으로 유지된다.

## 배포 전 최소 수동 점검

- 배달 주문 결제 성공 1건
- 배달 주문 전액 환불 1건
- 테이블 첫 주문 1건
- 테이블 추가 주문 1건
- 관리자 주문 상태 변경 1건
- POS 등록 성공 1건
- 메뉴 동기화 또는 직접 메뉴 생성 1건
